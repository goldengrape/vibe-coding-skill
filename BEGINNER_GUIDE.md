# 给新手看的说明

Vibe Coding Skill 是给“我有一个想法，但不知道怎么让 AI 稳定做出来”的人用的。

你可以把它理解成一个 AI 项目教练。它不会一上来就写代码，而是先帮你确认几个关键问题：

- 这个东西给谁用？
- 用户最主要要完成什么事？
- 哪些功能现在要做，哪些以后再说？
- 做到什么程度算完成？
- 有没有登录、隐私、付费、数据保存、外部接口这些限制？

确认以后，它会生成一组短文档。它们的作用是让后续 AI 编码时不容易跑偏。

| 你看到的名字 | 文件名 | 作用 |
| --- | --- | --- |
| 想法说明 | `URD.md` | 记录你真正想要什么。 |
| 拆解方案 | `ADD.md` | 检查项目能不能拆成清楚的部分。 |
| 模块说明 | `MDD.md` | 说明每个部分负责什么。 |
| 验收清单 | `TDD.md` | 说明怎样判断做对了。 |
| 开发路线 | `RMD.md` | 说明 AI 应该先做什么、后做什么。 |
| 项目地图 | `TRACE.md` | 把需求、设计、测试和任务连起来。 |

你不需要学习这些缩写。只要告诉 AI 你的想法，然后回答它提出的几个问题。

## 一个推荐用法

```text
我想做一个____。请用 Vibe Coding Skill 帮我先整理需求，不要急着写代码。请用新手能理解的话提问。
```

## 它为什么不直接写代码？

因为很多 AI 写坏项目，不是代码能力不够，而是它不知道你真正想做什么，也不知道哪些功能不该做。

先写小文档，是为了减少返工：

- 少做你没要的功能；
- 少漏关键限制；
- 少把几个功能写成一团；
- 后续修改时知道该改哪里；
- 测试时知道什么叫“完成”。


## 项目初始化和 Git

在真正写功能前，AI 应该先把项目文件夹整理好。Python 项目默认使用 `uv` 管理依赖，并创建 `pyproject.toml`、`.gitignore`、`README.md` 和 `tests/`。

写代码时不要一次做太多。每完成开发路线里的一个小任务，就运行测试，提交一次 Git，然后开一个 PR。PR 通过检查后再合并到主分支。这样即使 AI 写错了，也比较容易退回上一个稳定状态。
